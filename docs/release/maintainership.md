# Maintainership Policy

This document describes the maintainership structure and responsibilities for Spanish Law Research.

## Maintainers

The project is maintained by:

- Project lead and primary maintainer
- Additional contributors as designated by the project lead

## Maintainer Responsibilities

### Code Review

- Review pull requests in a timely manner
- Provide constructive feedback
- Ensure code quality and adherence to standards
- Verify tests are adequate and passing
- Check documentation updates
- Validate attribution and provenance for legal sources

### Issue Triage

- Monitor and triage new issues
- Label issues appropriately (bug, feature, documentation, security)
- Assign issues to maintainers or contributors
- Set priority levels based on impact and scope
- Close issues that are out of scope or resolved

### Security

- Respond to security reports through GitHub private security advisories
- Coordinate security fixes and disclosures
- Keep dependencies updated
- Monitor for vulnerabilities
- Ensure security policies are current

### Dependencies

- Monitor dependency updates
- Assess security vulnerabilities in dependencies
- Update dependencies when necessary
- Test dependency updates thoroughly before release
- Document breaking changes from dependency updates

### Release Management

- Prepare release checklists
- Coordinate release deployments
- Test releases before deployment
- Monitor releases post-deployment
- Handle rollbacks if necessary

### Community

- Foster a welcoming community
- Enforce the code of conduct
- Respond to questions and discussions
- Guide new contributors
- Recognize and appreciate contributions

## Index Freshness

Maintainers are responsible for:

- Monitoring sync process health
- Responding to sync failure alerts
- Investigating sync divergence issues
- Ensuring corpus remains current
- Coordinating with upstream Legalize project if source structure changes

## Decision Making

### Scope Decisions

- Maintain adherence to v1 scope (anonymous, read-only, tools-only)
- Evaluate feature requests against project goals
- Decide when to expand scope for future versions
- Keep documentation current about what is in/out of scope

### Technical Decisions

- Approve architectural changes
- Evaluate new dependencies
- Decide on refactoring priorities
- Balance technical debt with new features

### Process Decisions

- Update contribution guidelines as needed
- Modify release processes when beneficial
- Adjust testing requirements based on project maturity
- Improve documentation based on community feedback

## Becoming a Maintainer

New maintainers may be added based on:

- Consistent, high-quality contributions
- Understanding of project goals and architecture
- Active participation in code review and issue triage
- Alignment with project values and community standards
- Availability and commitment to the project

The process for adding a maintainer:

1. Existing maintainers discuss potential new maintainer
2. Candidate is approached and expresses interest
3. Candidate is given temporary maintainer permissions
4. Performance is evaluated over a trial period
5. Full maintainer status is granted if trial is successful

## Stepping Down

Maintainers who wish to step down should:

- Provide notice to other maintainers
- Complete ongoing reviews and responsibilities
- Document any pending work or knowledge
- Transition responsibilities to other maintainers
- Announce to community if appropriate

## Conflict Resolution

Maintainers should:

- Discuss disagreements respectfully
- Seek consensus when possible
- Escalate to project lead for final decisions if needed
- Document decisions and rationale
- Maintain professional relationships

## Removal of Maintainers

Maintainers may be removed for:

- Repeated violation of code of conduct
- Persistent inactivity without notice
- Compromise of account or security credentials
- Actions that harm the project or community

Removal process:

1. Other maintainers discuss the situation
2. Maintainer is given opportunity to respond
3. Decision is made by consensus of other maintainers
4. Access permissions are revoked
5. Community is informed if appropriate

## Emergency Contacts

For urgent security or operational issues, maintainers should have established communication channels outside of GitHub for rapid response.
